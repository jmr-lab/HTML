# Lightweight mail-client demo

<div class="w-full"><div class="progressive-markdown-content">
  <div class="content-visibility-auto"><p>The <code>inbox.html</code> is a <strong>self‑contained, static web page</strong> that implements a very lightweight “mail‑client” UI.<br>
It is built with plain HTML, CSS (mostly custom variables) and a handful of Bootstrap assets (icons, grid utilities).<br>
All interactive behaviour – opening a folder, selecting a message, toggling themes, showing/hiding panels – is driven <strong>exclusively by <code>&lt;input type="checkbox"&gt;</code> and <code>&lt;input type="radio"&gt;</code> elements</strong> together with CSS selectors (<code>:checked</code>, sibling combinators, etc.).</p>
<p>Because there is no JavaScript, no server‑side processing and no external data fetch, the page renders instantly in the browser and the UI updates are pure DOM/CSS operations.</p>
<hr>
<p><strong>How it works</strong></p>
<div class="markdown-table-wrapper"><table node="[object Object]"><thead><tr><th>Area</th><th>Mechanism</th><th>What you see</th></tr></thead><tbody><tr><td><strong>Navigation &amp; layout switches</strong></td><td>Hidden radio buttons (<code>#pHidden</code>, <code>#pVisible</code>, <code>#layoutOptions</code>, <code>#themeOptions</code>, <code>#cDark</code>, etc.) paired with <code>&lt;label&gt;</code> elements that act as clickable buttons. When a label is clicked the associated input becomes <code>:checked</code>. CSS rules that depend on <code>[id="…"]:checked ~ …</code> then show or hide sections, animate panels, or change CSS custom properties.</td><td>• Sidebar can collapse/expand. • Mail list can be displayed on the right, at the bottom, or hidden entirely. • Light/Dark/colour themes are applied instantly.</td></tr><tr><td><strong>Folder / message selection</strong></td><td>Each message row is a <code>&lt;label&gt;</code> linked to a hidden radio (<code>#content-001</code>, <code>#content-002</code>, …). Selecting a row checks the corresponding radio, which makes the matching <code>&lt;div id="mail‑001"&gt;</code>, <code>&lt;div id="mail‑002"&gt;</code>, … become visible while the “empty‑mail” placeholder disappears.</td><td>Clicking a line in the left pane opens the full message view on the right (or below, depending on layout).</td></tr><tr><td><strong>Action bar</strong></td><td>Buttons such as <strong>Create</strong>, <strong>Reply</strong>, <strong>Delete</strong> are ordinary <code>&lt;a&gt;</code> or <code>&lt;label&gt;</code> elements. Most are disabled (<code>class="disabled"</code>), but the <strong>Create</strong> button toggles the “new‑mail” form via <code>for="content‑new"</code>.</td><td>Users can start composing a new message; the form is shown/hidden without any script.</td></tr><tr><td><strong>Theme handling</strong></td><td>A set of CSS custom properties (<code>--ma-theme-bg</code>, <code>--ma-theme-color</code>, …) define colours for backgrounds, borders, text, etc. The theme‑radio inputs (<code>#tLight</code>, <code>#tDark</code>, <code>#tBlue</code>, …) each set a different group of these variables. The dark‑mode toggle (<code>#cDark</code>) swaps the whole palette.</td><td>Switching a theme instantly recolours the entire UI (sidebar, panels, buttons, text).</td></tr><tr><td><strong>Responsive design</strong></td><td>Media queries (<code>@media screen and (max-width: 700px)</code>) adjust the flex direction, hide the sidebar on narrow screens, and make the mail list take the full width. The layout still works because the same checkbox/radio logic applies regardless of screen size.</td><td>On phones/tablets the UI collapses to a single‑column view; on desktop you get the classic two‑pane Outlook‑like arrangement.</td></tr><tr><td><strong>Styling</strong></td><td>Uses a mixture of Bootstrap utility classes (<code>d-flex</code>, <code>flex‑column</code>, <code>list‑group</code>, <code>btn</code>, …) and a large block of custom CSS that defines colour variables, animations for modals/backdrops, scroll‑bar styling, and the visual look of the “mail‑icon” circles.</td><td>Looks fairly modern – icons, rounded avatars, hover highlights, subtle animations – comparable to a stripped‑down Outlook web view.</td></tr></tbody></table></div>
<hr>
<p><strong>What it actually does</strong></p>
<ol>
<li><strong>Displays a static mailbox UI</strong> – a sidebar with icons (Inbox, Outbox, Drafts, Sent, Spam, etc.) and a list of sample messages.</li>
<li><strong>Allows the user to open a message</strong> – clicking a row reveals the corresponding message body (hard‑coded Lorem ipsum text) in the right‑hand pane.</li>
<li><strong>Provides a “compose new mail” form</strong> – the form contains fields for From, To, Subject and Body, but it never sends anything; it’s purely visual.</li>
<li><strong>Offers theme and layout controls</strong> – the user can switch between many colour schemes, turn dark mode on/off, and choose whether the mail list appears on the right, at the bottom, or hidden.</li>
<li><strong>Implements all UI state with pure HTML/CSS</strong> – no JavaScript, no network calls, no backend. Because the browser only needs to apply CSS selectors, the interaction is essentially instantaneous.</li>
</ol>
<hr>
<p><strong>Why it can feel “super fast” compared with a full‑featured client like Outlook</strong></p>
<div class="markdown-table-wrapper"><table node="[object Object]"><thead><tr><th>Factor</th><th>Explanation</th></tr></thead><tbody><tr><td><strong>No JavaScript execution</strong></td><td>The browser never runs a script loop; it just toggles the <code>checked</code> state of native form controls.</td></tr><tr><td><strong>No network latency</strong></td><td>All markup, styles and images are already present in the file; there is nothing to fetch from a server.</td></tr><tr><td><strong>Simple DOM tree</strong></td><td>The page consists of a few hundred elements; updating visibility via CSS is a cheap operation.</td></tr><tr><td><strong>CSS‑only animations</strong></td><td>Show/hide transitions are done with <code>@keyframes</code> that the compositor can run on the GPU, avoiding layout thrashing.</td></tr><tr><td><strong>Bootstrap utilities</strong></td><td>They provide ready‑made responsive grids without extra calculations.</td></tr></tbody></table></div>
<p>These characteristics give a snappy, “instant‑response” feel, especially on low‑end devices or slow connections.</p>
<hr>
<p><strong>Limitations (compared with a real mail client)</strong></p>
<div class="markdown-table-wrapper"><table node="[object Object]"><thead><tr><th>Limitation</th><th>Impact</th></tr></thead><tbody><tr><td><strong>Static data</strong></td><td>All messages are hard‑coded; there is no IMAP/SMTP integration, no real inbox synchronization.</td></tr><tr><td><strong>No persistence</strong></td><td>Changing a theme or reading a message is lost on page reload because the state lives only in the DOM.</td></tr><tr><td><strong>No accessibility enhancements</strong></td><td>Relying on hidden checkboxes for navigation can be confusing for screen‑reader users; proper ARIA roles are missing.</td></tr><tr><td><strong>Limited interaction</strong></td><td>Buttons such as Delete, Reply, Forward are disabled; even the Compose form does not submit anywhere.</td></tr><tr><td><strong>Scalability</strong></td><td>Rendering thousands of messages would still be fast, but the UI provides no virtual scrolling or lazy loading.</td></tr><tr><td><strong>Security / privacy</strong></td><td>Since there is no backend, the page cannot protect or encrypt real email content.</td></tr></tbody></table></div>
<hr>
<p><strong>Summary</strong></p>
<p><code>inbox.html</code> is a <strong>demo of a minimalist, responsive mail‑client UI built entirely with HTML and CSS</strong>. By leveraging hidden checkboxes/radios and CSS selectors, it mimics many of the interactions you’d expect from Outlook (folder navigation, message preview, theme switching, layout changes) while staying extremely lightweight—no JavaScript, no server calls, and immediate rendering.</p>
<p>It showcases how far you can push pure‑CSS interactivity for a polished look, but it remains a <strong>static prototype</strong>; to become a functional email client you would need to add a data layer (API calls, authentication, storage) and probably some JavaScript for richer behaviours and accessibility. Nonetheless, as a proof‑of‑concept it demonstrates that a responsive inbox UI can indeed be built with just checkboxes and CSS, delivering very fast load and interaction times.</p></div></div></div>
